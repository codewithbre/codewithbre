# Personas: Alternative Onboarding Lenses

## The Idea

The `onboard-project` command scans a repo and produces a project-level CLAUDE.md. The current version is generalist: it identifies architecture, patterns, and conventions and produces a balanced operating contract.

The proposal is to create persona variants that approach the same scan through a different philosophical lens. The core philosophy stays constant — task integrity, scope constraints, stopping behavior. What changes is the emphasis, the questions asked during the scan, and the context surfaced in the output.

Same codebase. Different lens. Different CLAUDE.md.

---

## Possible Personas

**Security-first (`/onboard-security`)**
Scans for authentication patterns, data handling, trust boundaries, and known vulnerability surfaces. The resulting CLAUDE.md emphasizes what the model must never touch without a security review, and what questions to ask before modifying auth, sessions, or data flows.

**Scale and performance (`/onboard-scale`)**
Scans for bottlenecks, caching strategies, database access patterns, and architectural decisions made for throughput. The resulting CLAUDE.md establishes performance-sensitive zones and prevents the model from introducing patterns that would degrade them.

**Developer experience (`/onboard-dx`)**
Scans for code conventions, naming patterns, test structure, and team-specific idioms. The resulting CLAUDE.md is about consistency and contribution — onboarding a new developer (or a new agent) into how this team writes code.

**Legacy and migration (`/onboard-legacy`)**
Scans for technical debt, deprecated patterns, and migration state. The resulting CLAUDE.md treats existing patterns as intentional constraints — do not replace without a migration plan — and flags areas that are in transition.

**Agentic pipeline (`/onboard-agentic`)**
Scans for natural task boundaries, automation candidates, and handoff points. The resulting CLAUDE.md is optimized for multi-agent workflows: it identifies what can be parallelized, what must be sequential, and what requires human checkpoints.

---

## Format Options

Three approaches to how these personas exist in the system:

1. **Separate commands**: `/onboard-security`, `/onboard-dx`, etc. Each is a standalone command with its own prompt. Simple to implement, easy to discover, but creates duplication in the shared scanning logic.

2. **Parameterized command**: `/onboard-project --lens security`. One command, different behavior per lens. Cleaner, but requires the base command to support parameters and branch on them.

3. **Composable templates**: A base CLAUDE.md template generated by `/onboard-project`, with lens-specific overlays applied on top. The most flexible approach and the most aligned with the layered config architecture already in place.

Option 3 maps naturally to the existing global/project/local inheritance model. A persona overlay is just another configuration layer.

## Open Questions

- Should personas be additive (add constraints on top of the base) or substitutive (replace sections with lens-specific versions)?
- Is a persona a one-time scan output or a persistent configuration identity for the project?
- How do you handle a project that needs multiple lenses? (e.g., a legacy security-sensitive codebase)
- Should personas be user-defined, or is the set of personas fixed?
